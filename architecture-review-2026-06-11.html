1. ConvergeAnalysis 巨模块：拆分选址分析页面

src/views/ConvergeAnalysis.vue

Strong in-process
ConvergeAnalysis.vue (1500+ 行)

Before — 单文件承载一切

模板: 地图+工具栏+图例+控制面板 (300行)
script: 状态+turf分析+AMap渲染+弹窗+模拟 (900行)
style: 全局CSS+AMap穿透+组件样式 (500行)

一个 SFC 承担 8 种职责：地图初始化、网格分析、设施渲染、评分计算、弹窗生成、图层切换、分级筛选、选址模拟。修改任一部分都需要导航到这 1500 行中定位。

After — 按 seam 拆分为 5 个 module

ConvergeAnalysis.vue — 仅布局+编排 (80行)
useSpatialAnalysis.js — turf 网格计算接口
useMapLayers.js — AMap 图层渲染接口
useSiteScoring.js — 选址评分纯逻辑
ControlPanel.vue — 侧边栏参数面板

每个 module 拥有小而深的 interface：useSpatialAnalysis 暴露 { executeAnalysis(bbox) → gridResults }；useMapLayers 暴露 { renderGrid, renderFacilities, toggleLayer }。

Problem

一个 Vue SFC 承担 8 种正交职责。界面等于实现——每个改动都要在 1500 行中定位，没有独立的测试面。

Solution

沿自然 seam 拆分：空间分析、地图渲染、选址评分各成一个 composable hook，每个拥有可独立测试的 interface。

Wins

  • locality: 网格算法修改只需动 useSpatialAnalysis.js
  • leverage: useSiteScoring 可在无地图环境下纯逻辑测试
  • interface shrinks: 每个 hook 暴露 ≤5 个函数，替代当前 900 行 script
  • testability: turf 计算 + 评分逻辑可脱离浏览器运行

  • two adapters justify the seam: 真实 AMap + 测试 mock 两个 adapter
  • leverage: 新增地图页面只需调 useAmap()，不再拷贝初始化代码

8. 选址模拟污染 JSON 数据源：facilityData.push 跨路由持久化

ConvergeAnalysis.vue · facilities.json

Strong in-process
ConvergeAnalysis.vue:161ConvergeAnalysis.vue:600-604ConvergeAnalysis.vue:624

数据流污染路径

          flowchart LR
            FJ[facilities.json] -->|import| FD[facilityData]
            SM[选址模拟点击] -->|push| FD
            FD -->|污染| GA[网格分析]
            CA[撤销操作] -->|pop| FD
            style FD fill:#fecaca,stroke:#dc2626,stroke-width:2px
            style SM fill:#fecaca,stroke:#dc2626

第 600 行 facilityData.push({...}) 直接修改 import 的模块默认导出。Vite 下模块是单例，数据在路由切换后不重置。用户切走再切回，上次模拟的 "拟建" 设施仍然存在。第 624 行的 facilityData.pop() 依赖与 marker 数组同步，任何不一致都会导致数据漂移。

Problem

import 的 JSON 数据被 push/pop 直接变异。路由切换不重置，造成跨页面数据污染。没有 onBeforeUnmount 清理。

Solution

创建 ref([...facilityData]) 本地副本，模拟操作只修改副本。组件卸载时自动回收。

Wins

  • 数据完整性：原始 JSON 数据保持不变
  • 路由隔离：每次进入 /coverage 获得干净数据
  • adapter ready：本地副本模式天然支持后续 API 数据源切换

Top recommendation

#1: 拆分 ConvergeAnalysis 巨模块

这是当前代码库最大的架构风险。一个 1500+ 行的 SFC 承担了 8 种正交职责，修改任一部分都需要在全文中定位。 提取 useSpatialAnalysis / useMapLayers / useSiteScoring 三个 hook 是投入产出比最高的操作： 每个新 module 的 interface 小而深，可独立测试，后续增改功能只需动对应 hook。 这项拆分也是其他优化的前置条件——没有它，候选人 #5（数据层 seam）和 #2（AMap adapter）无处安放。

建议顺序：① ConvergeAnalysis 拆分 → ② AMap adapter → ③ 数据层 seam → ④ 常量提取 → ⑤ useDashboardState 分组
